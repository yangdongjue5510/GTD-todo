## 1. 프로젝트 배경 및 필요성

### 프로젝트 시작 배경

GTD-TODO 프로젝트는 기존 TODO 관리 도구들의 한계를 경험

기존 도구의 문제점: 

1. **프로젝트 단위 티켓 관리의 부재**: 칸반 보드에서 여러 티켓들을 관리할 때, 특정 티켓들이 하나의 프로젝트에 속한다는 것을 시각적으로 파악하기 어려움
2. **지속적인 사용의 어려움**: 바쁜 일상 속에서 일정 관리 앱과 실제 상황을 동기화하는 것을 잊게 되고, 이것이 반복되면 앱 사용 자체를 포기하게 됨

본 프로젝트는 개인 사이드 프로젝트로서, 다음과 같은 목적을 포함:
- AI를 활용한 소프트웨어 개발 프로세스 구축 및 검증
- Go 언어 기반 백엔드 개발 경험
- 바닐라 JavaScript 기반 프론트엔드 개발 경험
- 기획부터 개발, 배포까지 전체 제품 개발 사이클 경험

### 비즈니스 환경
- 개인 사이드 프로젝트 (비상업적)
- 비용 제로 또는 최소화된 인프라 운영
- 학습 및 경험 중심의 프로젝트

## 2. 타깃 유저 및 Pain Point

### 타깃 유저 (1차)
- 프로젝트 오너 본인
- 특성:
  - 여러 TODO와 프로젝트를 동시에 관리해야 하는 개인
  - 바쁜 일상으로 인해 일정 관리 도구 사용이 불규칙적
  - 기존 TODO 관리 도구의 한계를 경험한 사용자

### 주요 Pain Point

#### 1. 프로젝트 가시성 부족
- **문제**: 칸반 보드의 티켓들 중 어떤 티켓이 어떤 프로젝트에 속하는지 한눈에 파악하기 어려움
- **영향**: 프로젝트 단위 작업 관리 및 우선순위 설정의 어려움

#### 2. 일정 관리 앱 사용 중단
- **문제**: 바쁜 일상 속에서 앱과 실제 상황의 동기화를 잊게 됨
- **악순환**: 동기화 누락 → 수동 동기화 시도 → 반복적 실패 → 앱 사용 포기
- **영향**: TODO 관리 시스템의 붕괴, 생산성 저하

## 3. 프로젝트 성공 기준

### 학습 목표
- [ ] AI 기반 개발 프로세스 구축 완료
- [ ] AI를 활용한 개발이 기존 방식 대비 효율적임을 검증
- [ ] Go 언어 백엔드 개발 역량 확보
- [ ] 바닐라 JavaScript 프론트엔드 개발 역량 확보
- [ ] 전체 제품 개발 사이클 (기획-개발-배포) 완주

### 제품 목표
- [ ] 의도한 모든 핵심 기능 구현 완료
- [ ] 프로젝트별 티켓 관리 기능이 칸반에서 시각적으로 명확하게 표현됨
- [ ] "일정 관리 앱을 안 들어가게 되는 문제"에 대한 명확한 해결 시도가 기능에 반영됨
- [ ] 공개 배포 완료 (무료 인프라 활용)
- [ ] 기능 데모가 가능한 수준의 안정성 확보

### 사용자 목표
- [ ] 프로젝트 오너가 실제로 지속적으로 사용
- [ ] 기존에 겪었던 pain point가 실제로 해결됨

### 배포 목표
- [ ] Public 접근 가능한 상태로 배포
- [ ] 비용 제로 또는 최소화 (무료 티어 활용)
- [ ] 일정 기간 동안 안정적으로 운영 가능

## 4. 핵심 가설

1. **프로젝트 가시성 가설**: 칸반 보드에서 프로젝트별 티켓을 시각적으로 구분할 수 있으면, 작업 관리의 명확성이 증가할 것이다.

2. **지속 사용 가설**: 일정 관리 앱과 실제 상황의 동기화 문제를 해결하면, 사용자가 앱을 지속적으로 사용하게 될 것이다.

3. **AI 개발 효율성 가설**: AI를 활용한 개발 프로세스는 기존 방식보다 빠르고 쉽게 제품을 개발할 수 있게 할 것이다.

## 5. 사용자 리서치 결과 요약 (2단계)

### 사용자 행동 패턴

**TODO 관리 현황**:
- 과거 도구: Notion, Trello 사용 경험
- 사용 패턴: 앱 사용 시작 시 상시 확인, 아이디어 발생 시 Inbox에 추가
- 중단 원인: 업무 과중, 중압감 상황에서 앱 확인 중단 → 현실과 앱 상태 불일치 → 동기화 포기

**프로젝트 관리 현황**:
- 동시 진행 프로젝트: 2~3개
- TODO 규모: 최대 100개 미만
- 단발성 TODO도 다수 존재

### 핵심 인사이트

1. **심리적 부담 최소화 필요**
   - "현실과 앱 상태 동기화"가 가장 큰 사용 중단 원인
   - 정리를 강제하지 않고, 나중에 정리할 수 있는 유연성 필요

2. **프로젝트 가시성이 핵심**
   - 칸반 보드에서 프로젝트별 구분 필요
   - 특정 프로젝트 집중 모드 필요

3. **GTD 방법론 적합성**
   - Inbox 중심 접근이 사용자 패턴과 일치
   - 단계적 정리 워크플로우가 심리적 부담 감소에 효과적

### 차별화 전략

**MVP (Phase 1)**:
- GTD 전체 프레임워크 구현
- 프로젝트 배지 + 필터 기능으로 가시성 해결
- Inbox 빠른 추가로 심리적 부담 감소

**Phase 2** (향후 계획):
- 페르소나 기반 사용자 분석
- AI/ML 기반 추천 및 우선순위 제안
- 심리적 동기 부여 메시지 시스템

### 참조 문서

- [사용자 여정 맵](./user_journey_map.md): 상세 시나리오 및 워크플로우

## 6. 기능 정의 및 우선순위 (3단계)

### 기능 카테고리

GTD-TODO의 기능은 다음 5개 카테고리로 분류됩니다:

1. **Core TODO 관리**: GTD 워크플로우의 핵심 기능
2. **프로젝트 관리**: Pain Point 1 해결 (프로젝트 가시성)
3. **칸반 보드 UI**: 직관적인 사용자 인터페이스
4. **GTD Reflect 지원**: 주간 리뷰 및 정리 기능
5. **기본 기능**: 인증 및 데이터 영속성
